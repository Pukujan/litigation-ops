## What this project is

A litigation intelligence system built to capture, structure, and make searchable the institutional knowledge hidden inside a law firm's operational history.

The core problem: a mid-size NY litigation firm holds enormous operational intelligence — which clerks to call, how each judge behaves, what eCourts labels actually mean in practice, which courtrooms map to which parts, what to do when court orders have typos — but all of it lives in individual people's heads and buried inside case files. When those people are not available, the firm flies blind.

This system changes that by treating every operational incident as structured data, extracting intelligence from it automatically, and making it queryable by any attorney in real time.

## What happened today

Today was the full architecture session. No code written yet — this was the thinking session that everything else will be built from.

### The incident type that started this

The session opened with a real incident involving a Final Conference (NI-FCP) appearance scheduled in Kings County Supreme Court in a medical malpractice matter. The problem was not the appearance itself. It was that:

- No confirmed judge assignment in eCourts
- Two possible judges — each associated with a different courtroom
- General part rule pointed to one room, judge-specific standing order pointed to another
- Three possible clerk contacts with no clear routing logic for which to call first
- A consent order submission should waive appearance — but judge-specific rules may override that assumption

Resolution required calling the correct part clerk directly, with a calendar clerk as fallback. The answer existed — but finding it depended entirely on one person knowing the right sequence of calls to make.

This is the core structural problem: court data exists in fragments across eCourts, part rules, and judge-specific standing orders, with no authoritative tiebreaker and no system to resolve the conflict.

## Incidents reviewed and classified

Ten incident documents were reviewed in this session spanning multiple NY courts and operational domains. They revealed the full scope of what the system needs to handle.

### Incident domains identified

| Domain | Description | Example |
|---|---|---|
| Court ops | Judge/room/clerk ambiguity, eCourts mislabels, appearance uncertainty | NI-FCP judge unclear, wrong courtroom risk |
| Court order errors | Typos in orders creating hard compliance conflicts | EBT date error with fixed-schedule producing party |
| Case strategy | MSJ outcomes, expert strategy, liability chain analysis | Primary defendants dismissed, case structure collapses |
| Client management | Settlement expectation misalignment, communication breakdown | Low offer plus long payout timeline triggers client hostility |
| Firm operations | Manual workflows, repetitive low-value tasks | Court notice links requiring manual form retrieval |
| OPA / negotiation | Opposing counsel behavior, negotiation dynamics | Non-appearance as deliberate delay strategy |

### Risk layers identified

Every incident regardless of domain maps to one or more of three risk layers:

- **Operational** — wrong room, wrong clerk, wrong contact, wasted appearance
- **Procedural** — sanctions, defaults, missed deadlines, adverse rulings
- **Decision** — do I appear? do I file? do I call? which rule applies?

### Universal incident structure

Every incident regardless of type follows the same spine:

```
TRIGGER → CONDITION → ACTIONS TAKEN → OUTCOME → LESSON
```

The database is built around this structure.

## Key patterns observed across incidents

**Pattern 1 — eCourts is not reliable as sole source**
Labels can misrepresent actual procedural posture. Status entries can be administrative control dates that require no action. Future judge assignments are not confirmed. Calling the part clerk directly consistently produces faster and more accurate answers than relying on the system alone.

**Pattern 2 — General rules vs judge-specific rules conflict regularly**
Kings County is the clearest example — general Discovery Part rules set default courtrooms, but individual judges operate from different rooms. No hierarchy of authority is clearly documented. The conflict must be detected and flagged rather than silently resolved.

**Pattern 3 — Judicial behavior is a risk variable**
Different judges have meaningfully different enforcement styles. Strict enforcement around scheduling means a court order error is a critical risk, not an administrative nuisance. Informal resolution style means a phone call to chambers resolves what a letter cannot. This belongs in the database.

**Pattern 4 — Litigation timelines are often negotiated before they are filed**
Many deadline extensions, adjournments, and return date changes happen through informal OPA negotiation before any court involvement. This negotiation layer is invisible to rule-based systems but determines actual operational reality.

**Pattern 5 — Record volume does not correlate with case strength**
A case with thousands of pages of records may have its entire liability question concentrated in a two-hour window. Strategic narrowing of theory — and decisions about when to invest in expert retention — depend on recognizing this early.

**Pattern 6 — Experience, not systems, currently drives outcomes**
The difference between an efficient resolution and a missed appearance is consistently determined by institutional knowledge and prior experience — not anything a current system surfaces. This is the gap being built.

## What existing tools do and don't do

**Westlaw / Lexis / e-Law** — publish what rules say, not how courts behave. No institutional memory from real firm experience.

**E-Discovery tools** — document review and production. Entirely different layer.

**Clio / Litify / MyCase** — practice management. Track that something is due, not how to handle ambiguity when it arrives.

**Legora / Harvey / CoCounsel** — AI research and drafting on published legal content. Not operational intelligence.

**Opus 2 / Docket Alarm** — closest to this space. Static rules only. No incident layer, no judge behavior profiles, no clerk routing logic.

**What no tool captures:**
- That eCourts labels can be wrong and calling the part clerk directly is faster
- That certain counties accept late submissions informally while others require confirmed receipt
- That a new judge assignment can mean same-day scheduling with hours of notice
- That opposing counsel non-appearance at a conference is sometimes a deliberate delay strategy
- That a strict judge treats a court order typo as a compliance failure regardless of good faith

## System flowchart

The full AI ingestion and storage flow was designed today. The sequence:

1. **Input** — incident reports, part rule summaries, case history snapshots, best practice notes (any format)
2. **AI agent ingestion** — reads raw document, detects structure
3. **Document type detection** — incident? rule summary? case history? best practice note?
4. **Domain classification** — court ops / case strategy / client mgmt / firm ops / OPA negotiation
5. **Risk layer tagging** — operational / procedural / decision
6. **Structured extraction** — court, judge, case stage, contacts, rules, trigger, actions, resolution, lesson
7. **Conflict detection** — new data vs existing reference layer, flag uncertainty if contradicting
8. **Three-layer storage** — incident log (immutable) + update log (append-only) + reference layer (living)
9. **Attorney-facing output** — query by court, judge, case stage, or incident type

The reference layer enriches future classifications — the system gets smarter with every incident logged.

## Incident report template — what changed and why

### What raw incident reports already had

The incident documents reviewed today were already rich — far better than typical firm notes. They contained full appearance history tables, named contacts with confirmed information, chronological action timelines with timestamps, root cause and structural analysis, and reusable IF/THEN operational rules written out explicitly.

### First template attempt — what went wrong

The first template compressed the narrative into short fillable fields. This was wrong. The detail is the value. A compressed incident report loses the named contacts, the failure points, the causal chains that make it intelligence rather than just a note.

### Corrected structure

```
ANCHOR BLOCK     — 8 short machine-readable fields (AI grabs these first)
     ↓
FULL NARRATIVE   — write everything, exactly as detailed as the raw reports
     ↓
LESSON BLOCK     — IF/THEN rule, pattern tag, reference layer flag (AI extracts last)
     ↓
UPDATE LOG       — append-only timestamped entries, never overwrite
```

The anchor block and lesson block give the AI clean entry and exit points. Everything in the middle stays as rich and detailed as naturally written.

**Key insight:** the natural writing style in these reports — detailed causal analysis, named people, failure documentation — is an asset, not something to normalize away. The template wraps it, not replaces it.

## Database schema — 3 layers, 18 tables

### Technology decision: PostgreSQL

Chosen over NoSQL because the data is relational, SQL handles cross-relationship queries cleanly, and the JSONB column type handles court-specific edge cases without breaking the structure. Hosted on Supabase for the initial build.

### Layer 1 — Court reference

| Table | Purpose |
|---|---|
| `courts` | One row per court system. The universe separator — all other data hangs off this |
| `judges` | Judge profiles, behavior summaries, part assignments |
| `parts` | Part names, types, default courtrooms, appearance rules |
| `court_contacts` | Named clerks, phones, emails — traceable to the incident that verified them |
| `court_rules` | General and judge-specific rules, with conflict flag if they contradict |

### Layer 2 — Cases and incidents

| Table | Purpose |
|---|---|
| `cases` | Index number is the universal key connecting everything |
| `case_appearances` | Court appearance history per case |
| `incidents` | Core table. Immutable raw narrative plus all extracted structured fields |
| `incident_updates` | Append-only resolution log — every action taken, every confirmation received |
| `incident_participants` | Named people involved — feeds back into court_contacts when verified |

### Layer 3 — Knowledge and intelligence

| Table | Purpose |
|---|---|
| `lessons_learned` | Reusable IF/THEN rules derived from incidents, with scope and confidence score |
| `conflict_flags` | Stores both sides of a contradiction without resolving prematurely |
| `judge_profiles` | Behavioral observations accumulate over time into confident profiles |
| `ecourts_flags` | Documents known eCourts label problems and correct interpretations |
| `ingestion_log` | Tracks every document the AI processed — full audit trail |

### Key design decisions

- `incidents.raw_narrative` stores the original report text untouched — immutable source of truth
- Index number on the case table connects NYSCEF, incidents, and case management without touching PII
- `conflict_flags` stores both sides of a contradiction without resolving — resolution happens when confirmed by a real source
- `judge_profiles.observations_count` grows with each confirming incident — one observation is a note, ten is institutional knowledge
- `incident_participants` is its own table (not JSONB) because named contacts are valuable reference data worth querying

## Data sources

### Phase 1 — Forward-looking (build now)
- New incident reports ingested as written going forward
- NYSCEF public records pulled by index number
- Manual court contact verification
- No PII, no privilege issues

### Phase 2 — Backward-looking (later, carefully)
- Existing case files live in the firm's case management system
- Contains privileged communications, attorney notes, client data — cannot be touched directly
- Approach: human extracts only non-sensitive operational facts, or controlled pipeline with proper access controls
- The index number connects existing cases to the operational layer without exposing sensitive data

## Key architectural decisions

1. PostgreSQL over NoSQL — relational structure fits, JSONB handles edge cases
2. Court as universe separator — each court system isolated, no cross-contamination
3. Incident as immutable source of truth — raw narrative never overwritten
4. Index number as universal key — connects data layers without exposing PII
5. Three-layer storage — reference (stable) / incident (immutable) / knowledge (living)
6. AI detects document type before extracting — routes to correct tables
7. Conflict detection before storage — contradictions flagged, not silently resolved
8. Judge profiles accumulate over time — confidence grows with observations
